Introduction

Internet, Web et applications web sont des mots que nous entendons tous les jours mais qui sont pourtant sujets à de nombreuses confusions. Il semble primordial d’en donner une définition précise.

Internet est un réseau de réseaux à l’échelle mondiale. Il permet l’échange de paquets à travers ce réseau. Le Web n’est qu’une application d’Internet et permet de consulter des pages accessibles sur des sites à l’aide d’un navigateur. Enfin, les applications web sont des applications hébergées par des serveurs et accessibles par des navigateurs web. Ces applications sont nombreuses et variées : réseaux sociaux, webmail, sites de e-commerce, jeux. Il convient donc de sécuriser le Web qui contient des données sensibles, confidentielles ou personnelles.

De nombreux acteurs aident à se prémunir contre les attaques des hackers. En France, l’ANSSI qui s’occupe de la sécurité des systèmes d’information met à disposition des guides sur les bonnes pratiques pour la sécurité des systèmes informatiques. Aux Etats-Unis, l’OWASP est une communauté de bénévoles dédiée à la sécurité des applications web. Elle publie des articles, guides et logiciels pour sensibiliser aux risques liés à la sécurité web. De même, le WASC qui rassemble des experts en sécurité, publie des documents sur la sécurité web.

Architecture d’internet et fonctionnement du Web

Le Web repose sur le réseau internet, il convient donc d’en décrire brièvement le fonctionnement. Internet est construit en différentes couches. Le modèle majoritairement utilisé est le couple TCP/IP qui fonctionne en quatre couches : la couche accès réseau, la couche internet, la couche transport et la couche applicative.

Le Web utilise toutes ces couches pour ses échanges de données. Ce qui distingue les différents services d’Internet est le protocole utilisé par la couche applicative. Par exemple, pour échanger des mails le protocole SMTP est utilisé. Le Web fonctionne grâce au protocole HTTP.

Le protocole HTTP a été développé pour le Web et a permis son développement. C’est un protocole de messages : l’utilisateur envoie une requête par son navigateur web et la requête est traitée par le serveur qui renvoie une réponse au client.

Le Web a été créé en 1989 et a connu une évolution importante.

L’évolution du Web : du Web 1.0 au Web 3.0

Web 1.0 (1990-2000) : le Web statique

Dans les années 90, le Web avait pour vocation d’afficher uniquement du contenu aux utilisateurs, c’est pourquoi on parle de Web statique. Les sites web pouvaient être vus comme des magazines. Ils étaient limités à des liens hypertextes et à du contenu multimédia.

Web 2.0 (2000-2010) : le Web participatif axé sur les utilisateurs

A partir des années 2000, le comportement des utilisateurs a changé et le Web est devenu collaboratif. L’internaute ne consulte plus uniquement, il peut maintenant publier du contenu et est sollicité à le faire. Le Web 2.0 voit apparaître de multiples réseaux sociaux ou blog comme Facebook et YouTube.

Web 3.0 (2010-20xx) : le Web sémantique ou le Web de l’Internet des objets

Avec la multiplication des téléphones mobiles, tablettes et objets connectés, le Web a dû évoluer pour se connecter au monde réel des utilisateurs. Les maisons connectées sont un très bon exemple, le chauffage peut alors s’adapter à la météo. Ce Web est également centré sur l’utilisateur, ce n’est plus l’utilisateur qui agrège les données mais des suggestions viennent à lui basées sur son historique ou ses données personnelles. Cette évolution implique une évolution des technologies et de potentielles incompatibilités entre elles. Pour guider les différentes technologies du Web dans une même direction, le W3C pour World Wide Web Consortium est une communauté internationale qui s’occupe de développer les normes du Web.

Pourquoi sécuriser son site web ?

Le Web n’est plus uniquement utilisé pour afficher du contenu statique aux utilisateurs. Il présente différentes applications : moteurs de recherche, webmail qui permet de gérer ses mails par une interface web, le commerce électronique et les jeux en ligne, les réseaux sociaux. Toutes ces applications peuvent être sujettes à des actes de piratage.

Il n’est pas rare de voir dans les actualités des cas de piratage informatique. Par exemple en 2016, le site de e-commerce d’Acer a été victime d’une attaque de hackers qui ont réussi à dérober les informations personnelles de 34 500 clients incluant leurs coordonnées bancaires. En 2017, Anonymous a saturé le site de la Cour Constitutionnelle espagnole pour montrer leur soutien quant à l’indépendance de la Catalogne. Des attaques ont aussi été menées sur le site de la police espagnole. Ces exemples montrent que les attaques web sont utilisées pour différentes fins et ne sont pas rares. La sécurisation des sites web est donc un enjeu majeur.

La sécurité web

La sécurité web est la pratique de protéger les sites web contre l'accès, l'utilisation, la modification, la destruction ou la perturbation non autorisés. La sécurité ne s’arrête pas à la prévention mais aussi à la réaction de la victime en cas d’attaque.

1. Prévention

De nombreux acteurs comme l’OWASP ou l’ANSSI ont rédigé des documents pour prévenir les attaques web. Ces préventions se font au niveau du développement et du système hébergeant le service.

Développement

L’OWASP a dressé une liste des bonnes pratiques de développement rassemblées dans un livret de 17 pages. Ce guide doit permettre aux développeurs de produire du code qui n’est pas vulnérable aux attaques les plus courantes. L’ANSSI a également rédigé une note technique de bonnes pratiques de développement. Ce document est plus complet que celui de l’OWASP dans le sens où il couvre à la fois la prévention pour la sécurité web mais aussi les bons réflexes à avoir en cas d’attaques.

Tests

Tout développement se fait en parallèle avec des tests. Encore faut-il savoir quoi, pourquoi, comment, quand et où tester les applications web. L’OWASP a rédigé un guide de tests qui permet de répondre à ces questions. Ce guide propose même une structure complète de tests pour que le lecteur puisse créer son propre programme de tests. L’objectif de ce document est de guider le lecteur dans ses choix de techniques de tests afin de produire des applications web fiables et sécurisées.

Infrastructure

Un travail est aussi à faire au niveau de l’architecture et de la configuration du système qui héberge le service. La note technique de l’ANSSI traite de ce sujet en expliquant les principes de base à suivre.

Scanners et logiciels préventifs

Un grand nombre de logiciels de scan de sites web sont disponibles. Ces logiciels permettent de détecter les vulnérabilités des applications web. Le WASC a rédigé un guide, le WASSEC, pour évaluer les scanners d’application web en fonction de leur efficacité à tester les applications web et à trouver leur vulnérabilité. Le lecteur peut alors évaluer les différents scans proposés en fonction des critères du guide pour choisir le scanner qui répond le mieux à ses attentes.

Le WASC a également listé les scans payants et gratuits les plus utilisés. Plusieurs logiciels de l’OWASP y figurent. L’un des plus connus est le logiciel ZAP pour Zed Attack Proxy qui automatise la détection des failles et propose de nombreuses fonctionnalités telles que l’interception d’un serveur proxy, le scan actif ou passif d’un site web.

D’autres logiciels, qui ne sont pas des scanners, permettent aussi de faire de la prévention contre des attaques web. Webscarab par exemple, développé par l’OWASP, agit comme un serveur proxy et permet de modifier les requêtes HTTP avant qu’elles ne soient envoyées au serveur.

Audits de sécurité

Un audit de sécurité informatique est une expertise qui permet de relever les failles d’un système informatique. Un plan d’action pour contrer ces failles est ensuite établi. Des sociétés comme Vaadata ou le groupe GTS proposent de réaliser des audits de sécurité d’application web. Les tests d’intrusion sont la méthode la plus utilisée pour réaliser des audits de sécurité des applications web.

Ils sont classés en trois catégories :

  • Boîte noire : Les tests d’intrusion sont effectués en conditions réelles. L’auditeur réalise les tests de l’extérieur et ne possède pas d’informations supplémentaires sur le système testé.
  • Boîte grise : L’auditeur possède quelques informations sur le système, généralement un compte utilisateur lui est donné.
  • Boîte blanche : Les réglages mis en place sont connus par l’auditeur.

La deuxième étape des tests d’intrusion consiste à essayer d’utiliser les failles découvertes pour compromettre le système. Cette phase est appelée « l’exploit ». On retrouve aussi l’audit de code qui consiste à lire le code source d’une application web pour rechercher les failles.

2. Réaction

En cas d’attaque, l’organisme touché doit être capable de réagir rapidement et en conséquence. La note technique de l’ANSSI couvre ce point. Elle conseille d’avoir un point de contact associé au site facilement identifiable pour être prévenu rapidement en cas d’attaque.

De plus, pour réagir rapidement, l’attaque doit être détectée rapidement. L’ANSSI conseille de faire un travail de journalisation en sauvegardant par exemple le dernier échec de connexion des utilisateurs. La surveillance est également primordiale dans la sécurité web, parcourir son site à la recherche d’anomalie permet de détecter des attaques.

En cas d’incident, il est primordial de regrouper toutes les informations possibles pour permettre d’investiguer. Il est également indispensable de corriger les vulnérabilités du site avant de le remettre en service.

Les failles les plus courantes

L’OWASP recense tous les trois ans les dix failles web les plus courantes. Pour déterminer ce classement, l’OWASP se base sur trois critères : difficulté à trouver la faille, difficulté pour l’exploiter et le préjudice que l’attaque peut causer. Le top 10 de 2017 est disponible en suivant ce lien. Un résumé se trouve ci-dessous.

1. Injection

Les failles d’injection, telles que SQL, noSQL, OS, LDAP sont courantes dans les applications web. L’injection consiste à fournir une donnée malicieuse à un interpréteur dans le cadre d’une commande ou requête. La requête peut alors duper l’interpréteur et permet à l’attaquant d’accéder, de modifier ou de supprimer des données normalement protégées.

2. Violation de gestion d’authentification et de session

Les droits d'accès aux comptes et les jetons de session sont souvent incorrectement protégés. Les attaquants compromettent les mots de passe, les clefs, ou les jetons d'authentification pour s'approprier les identités d'autres utilisateurs.

3. Protection insuffisante des données sensibles

Beaucoup d’applications web et d’API ne protègent pas correctement leurs données sensibles. Des attaquants peuvent alors voler ou modifier ces données pour par exemple commettre des fraudes sur les cartes de crédit. Des protections supplémentaires telles que l’encodage sont donc nécessaires.

4. XML eXternal Entities : XXE

Un document XML se base sur une grammaire pour vérifier sa conformité. Cette grammaire est contenue dans un fichier appelé DTD (Document Type Definition). Des entités qui font références à des ressources externes peuvent être déclarées dans le DTD pour définir des variables. Ainsi en déclarant des entités bien choisies dans le DTD, l’attaquant peut accéder à des fichiers protégés.

5. Violation de contrôle d’accès

Les restrictions sur ce que les utilisateurs authentifiés ont le droit de faire ou non ne sont généralement pas mises en œuvre correctement. Les attaquants peuvent exploiter ces failles pour accéder à du contenu non autorisé.

6. Mauvaise configuration sécurité

Une mauvaise configuration de sécurité est l’erreur la plus courante. La configuration concerne l’ensemble suivant : l’application, le contexte, le serveur d’application, le serveur web, le serveur de données et la plate-forme. Cet ensemble doit être correctement configuré et mis à jour régulièrement.

7. Cross Site Scripting : XSS

L’injection de données arbitraires dans un site web peut révéler des failles si les données transmises ne sont pas vérifiées. Ces failles permettent d’exécuter des scripts dans le navigateur de la victime pour voler des sessions utilisateurs par exemple.

8. Désérialisation non sécurisée

La sérialisation consiste à convertir un objet dans un format qui est facilement stockable. La désérialisation est le processus inverse et permet de récupérer l’objet. On parle de désérialisation non sécurisée si les données à désérialiser peuvent été modifiées par un utilisateur sans vérification. La désérialisation non sécurisée entraîne souvent l’exécution de code à distance. Elles peuvent aussi être utilisées pour d’autres types d’attaques tels que les attaques par injection ou par rejeu.

9. Utilisation de composants avec des vulnérabilités connues

Les composants tels que les librairies, les frameworks et les autres modules logiciels ont les mêmes privilèges que l’application. Si un de ces composants présente une vulnérabilité, l’application est aussi compromise. L’utilisation de composants qui présentent des vulnérabilités connues peut alors donner lieu à des attaques sur l’application.

10. Insuffisance d’authentification et de contrôle

L’insuffisance d’authentification et de contrôle couplée à une mauvaise gestion des réponses suite à un incident est un terrain de jeu parfait pour les attaquants. Ils peuvent mener des attaques en toute discrétion.
