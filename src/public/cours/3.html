Les failles les plus courantes

L’OWASP recense tous les trois ans les dix failles web les plus courantes. Pour déterminer ce classement, l’OWASP se base sur trois critères : difficulté à trouver la faille, difficulté pour l’exploiter et le préjudice que l’attaque peut causer. Le top 10 de 2017 est disponible en suivant ce lien. Un résumé se trouve ci-dessous.

1.Injection

Les failles d’injection, telles que SQL, noSQL, OS, LDAP sont courantes dans les applications web. L’injection consiste à fournir une donnée malicieuse à un interpréteur dans le cadre d’une commande ou requête. La requête peut alors duper l’interpréteur et permet à l’attaquant d’accéder, de modifier ou de supprimer des données normalement protégées.

2.Violation de gestion d’authentification et de session

Les droits d'accès aux comptes et les jetons de session sont souvent incorrectement protégés. Les attaquants compromettent les mots de passe, les clefs, ou les jetons d'authentification pour s'approprier les identités d'autres utilisateurs.

3.Protection insuffisante des données sensibles

Beaucoup d’applications web et d’API ne protègent pas correctement leurs données sensibles. Des attaquants peuvent alors voler ou modifier ces données pour par exemple commettre des fraudes sur les cartes de crédit. Des protections supplémentaires telles que l’encodage sont donc nécessaires.

4.XML eXternal Entities : XXE

Un document XML se base sur une grammaire pour vérifier sa conformité. Cette grammaire est contenue dans un fichier appelé DTD (Document Type Definition). Des entités qui font références à des ressources externes peuvent être déclarées dans le DTD pour définir des variables. Ainsi en déclarant des entités bien choisies dans le DTD, l’attaquant peut accéder à des fichiers protégés.

5.Violation de contrôle d’accès

Les restrictions sur ce que les utilisateurs authentifiés ont le droit de faire ou non ne sont généralement pas mises en œuvre correctement. Les attaquants peuvent exploiter ces failles pour accéder à du contenu non autorisé.

6.Mauvaise configuration sécurité

Une mauvaise configuration de sécurité est l’erreur la plus courante. La configuration concerne l’ensemble suivant : l’application, le contexte, le serveur d’application, le serveur web, le serveur de données et la plate-forme. Cet ensemble doit être correctement configuré et mis à jour régulièrement.

7.Cross Site Scripting : XSS

L’injection de données arbitraires dans un site web peut révéler des failles si les données transmises ne sont pas vérifiées. Ces failles permettent d’exécuter des scripts dans le navigateur de la victime pour voler des sessions utilisateurs par exemple.

8.Désérialisation non sécurisée

La sérialisation consiste à convertir un objet dans un format qui est facilement stockable. La désérialisation est le processus inverse et permet de récupérer l’objet. On parle de désérialisation non sécurisée si les données à désérialiser peuvent été modifiées par un utilisateur sans vérification. La désérialisation non sécurisée entraîne souvent l’exécution de code à distance. Elles peuvent aussi être utilisées pour d’autres types d’attaques tels que les attaques par injection ou par rejeu.

9.Utilisation de composants avec des vulnérabilités connues

Les composants tels que les librairies, les frameworks et les autres modules logiciels ont les mêmes privilèges que l’application. Si un de ces composants présente une vulnérabilité, l’application est aussi compromise. L’utilisation de composants qui présentent des vulnérabilités connues peut alors donner lieu à des attaques sur l’application.

10.Insuffisance d’authentification et de contrôle

L’insuffisance d’authentification et de contrôle couplée à une mauvaise gestion des réponses suite à un incident est un terrain de jeu parfait pour les attaquants. Ils peuvent mener des attaques en toute discrétion.
